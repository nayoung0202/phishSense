# 현재 범위

이 문서는 현재 코드베이스가 제공해야 하는 기능 범위를 요약합니다. 상세 구현은 소스 코드가 아니라 이 문서를 기준으로 해석합니다.

## 기능 요구사항

### 1. 인증과 접근 제어

- OIDC 로그인과 콜백 처리를 지원해야 합니다.
- 라우트는 `public`, `session`, `ready` 세 수준으로 보호해야 합니다.
- 세션이 없는 페이지 요청은 로그인으로 리다이렉트하고, API 요청은 `401`을 반환해야 합니다.
- tenant 또는 entitlement가 준비되지 않은 사용자는 onboarding으로 유도해야 합니다.
- tenant와 entitlement가 준비된 사용자는 업무 데이터가 비어 있어도 제품에 정상 진입할 수 있어야 합니다.
- 업무 데이터가 비어 있는 ready tenant는 첫 체험 흐름 또는 운영 설정 동선으로 자연스럽게 진입할 수 있어야 합니다.

### 2. 플랫폼 연동

- `platform-api`의 `/platform/me`를 이용해 tenant/entitlement 컨텍스트를 조회해야 합니다.
- 첫 로그인 사용자는 제품 BFF를 통해 tenant를 생성할 수 있어야 합니다.
- 제품은 tenant 멤버 목록 조회, invite 생성, invite 수락 BFF를 제공해야 합니다.
- invite 수락 성공 후 `/platform/me`를 재조회해 tenant 컨텍스트를 갱신해야 합니다.
- `platform-api` entitlement callback을 수신해 로컬 DB 상태를 갱신해야 합니다.
- 제품 접근 허용의 최종 기준은 로컬 entitlement 상태여야 합니다.
- tenant 생성 직후의 업무 데이터 provisioning 자동화는 현재 단계의 필수 범위가 아닙니다.

### 3. 훈련 운영

- 프로젝트, 템플릿, 대상자, 교육 페이지 CRUD를 제공해야 합니다.
- 프로젝트, 템플릿, 대상자, 교육 페이지, 발송 설정 관리 목록은 페이지네이션을 제공해야 합니다.
- 대상자 등록과 대상자 엑셀 업로드는 활성 entitlement의 `seatLimit` 범위 안에서만 허용해야 합니다.
- AI 생성 템플릿/교육 페이지를 저장하고, 자기 자신에게 보내는 1인 체험 프로젝트를 만들 수 있어야 합니다.
- 프로젝트는 템플릿, 교육 페이지, 발송자 정보, 기간, 대상자와 연결되어야 합니다.
- 발송 큐와 워커를 통해 메일을 비동기로 전송해야 합니다.
- 오픈, 클릭, 제출 상태를 프로젝트 단위로 집계해야 합니다.
- tenant-owned 업무 데이터는 tenant 기준으로 논리적으로 격리되어야 하며 cross-tenant 접근이 차단되어야 합니다.
- 공개 token 라우트는 token-only URL을 유지하되, 서버 내부에서 관련 데이터의 tenant 일관성을 검증해야 합니다.

### 4. 보고서

- 보고서 설정과 템플릿을 관리할 수 있어야 합니다.
- 프로젝트 기준 보고서 생성을 수행하고 결과 파일을 다운로드할 수 있어야 합니다.
- 기본 보고서 생성은 `attached_assets/default_report_template.docx`를 기준 템플릿으로 사용해야 하며, 별도 `templateId`가 명시된 경우에만 tenant 저장소의 보고서 템플릿 파일을 사용해야 합니다.

### 5. 운영 설정

- 설정 허브는 언어, 멤버, 구독, 크래딧 섹션을 제공해야 합니다.
- 사이드바 최하단 계정 카드에서 현재 로그인 이메일, 설정 진입, 로그아웃을 제공해야 합니다.
- 언어 설정은 `ko`, `en`, `ja`를 지원하고 `ps_locale` 쿠키를 기준으로 동작해야 합니다.
- 언어 설정 변경 시 공통 셸, 로그인/온보딩, 설정, 보고서, 발송 설정을 포함한 주요 서비스 UI가 같은 locale 기준으로 즉시 다시 렌더링되어야 합니다.
- 구독 UI는 서버 카탈로그를 렌더링해야 하며 제품 클라이언트가 가격 정책의 source-of-truth가 되어서는 안 됩니다.
- 크래딧 차감 정책은 usage context를 포함한 플랫폼 credit authorization 흐름으로 제어해야 합니다.
- 초기 포함 크레딧은 tenant 생성 후 `platform-api`의 entitlement plan 확정 시점에 자동 부여해야 하며, `FREE`는 `3`, `BUSINESS`는 `10`을 기본값으로 사용해야 합니다.
- 크레딧 화면은 남은 크레딧, 포함 크레딧, 예약 차감, 차감 정책, 최근 변동, 충전 CTA를 제공해야 합니다.
- `체험하기` 흐름을 포함한 모든 AI 생성은 동일한 크레딧 차감 정책을 따라야 합니다.
- 테넌트별 발송 설정을 제공해야 합니다.
- 발송 설정은 SMTP 연결 정보, 허용 발신 도메인, 테스트 발송 기능을 포함해야 합니다.
- 발송 설정은 프로젝트 생성 화면에서 구분 가능한 사용자 지정 별칭을 지원해야 합니다.
- 허용 발신 도메인은 발송 설정당 최대 5개까지 등록할 수 있어야 합니다.
- 허용 발신 도메인 검증은 등록 도메인 자체와 그 하위 도메인을 모두 허용해야 합니다.
- 프로젝트 발신 이메일과 SMTP 테스트 발신 이메일은 선택한 발송 설정의 허용 발신 도메인 정책을 따라야 합니다.
- 템플릿 HTML은 발송 전 검증과 보정 절차를 거쳐야 합니다.
- AI 템플릿 생성은 1회당 `2`크레딧, AI 훈련 안내 페이지 생성은 1회당 `1`크레딧을 차감해야 합니다.
- AI 생성 요청 전에는 현재 잔액이 차감 기준을 충족하는지 확인해야 하며, 부족하면 생성 대신 바로 결제 기반 충전 동선을 제공해야 합니다.
- tenant-owned 파일 자산은 tenant prefix 기반 저장 경로 규칙을 따라야 합니다.

## 비기능 요구사항

- 데이터는 PostgreSQL에 영속 저장되어야 합니다.
- 스키마와 입력 검증은 TypeScript 타입과 Zod 스키마로 일관되게 관리되어야 합니다.
- 보안상 민감한 토큰과 비밀번호는 평문으로 저장하지 않아야 합니다.
- 주요 API, 인증 흐름, 플랫폼 callback은 테스트로 검증 가능해야 합니다.
- tenant 격리 전환은 기존 테스트를 전면 폐기하지 않고 회귀 테스트를 유지·보강하는 방식으로 진행해야 합니다.
